# 核心稳定性更新设计

## 目标

在不重构现有页面和整体分层架构的前提下，解决当前项目中最影响可运行性、安全性和数据准确性的核心问题，使前后端具备更可靠的本地开发基础。

## 范围

本次更新包含：

- 用户密码从固定盐 MD5 直接切换为 BCrypt。
- 数据库连接、JWT 和 AI 配置支持环境变量覆盖。
- 修复前端健康记录更新接口与后端不一致的问题。
- 修正健康指标平均值计算逻辑，仅使用实际填写的有效数据。
- 为分页、统计周期和导出格式增加参数边界校验。
- 为同一用户同一天的健康数据增加数据库唯一约束。
- 增加后端核心 Service 单元测试。
- 明确前端要求 Node.js 20 或更高版本。

本次不包含：

- 旧 MD5 密码迁移；现有测试用户允许清空后重新注册。
- UI 或交互改版。
- 全局异常体系和 HTTP 状态码重构。
- Flyway/Liquibase 数据库迁移框架。
- OpenAPI 文档和完整 CI/CD。
- AI 接口协议升级。

## 架构和组件变化

### 用户认证

后端引入 Spring Security Crypto 提供的 `BCryptPasswordEncoder`，但不引入完整 Spring Security 过滤器链。注册时使用 BCrypt 生成随机盐哈希，登录时使用 `matches` 校验密码。

该方案保持现有 JWT 拦截器、登录接口和前端登录流程不变，只替换密码存储实现。数据库 `password` 字段长度 128 足以保存 BCrypt 哈希。

### 配置管理

`application.yml` 使用 Spring 占位符读取环境变量，并提供仅适合本地开发的默认值。生产部署必须显式设置数据库密码、JWT 密钥和 AI Key。

计划使用的变量包括：

- `DB_URL`
- `DB_USERNAME`
- `DB_PASSWORD`
- `JWT_SECRET`
- `JWT_EXPIRATION`
- `AI_API_URL`
- `AI_API_KEY`
- `AI_MODEL`

前端 API 地址改为从 Taro 环境变量读取，未配置时保持本地 `http://localhost:9999/api` 默认值。

### 健康记录写入

继续保留后端“按用户和日期新增或覆盖”的接口语义：

```text
POST /api/health/record
```

前端删除无后端支持的 `POST /api/health/record/{id}` 调用，所有保存行为统一调用 `/health/record`。

数据库将 `(user_id, record_date)` 从普通索引调整为唯一索引，确保并发请求不能生成重复的每日记录。Service 仍保留现有查询后更新逻辑；数据库唯一约束作为最终一致性保护。

### 统计计算

每项平均值独立筛选有效数据：

- 步数、心率等整数指标：仅统计非空值。
- 睡眠、体重和血糖：仅统计非空值。
- 某项完全没有数据时返回 0。

趋势数组继续用 0 表示某一天没有填写，以保持现有图表数据结构不变。

### 参数校验

Controller 接口保持不变，Service 对外部参数执行边界校验：

- `page >= 1`
- `1 <= size <= 100`
- `1 <= days <= 365`
- 导出格式只接受 `csv` 或 `json`

无效参数抛出带有明确中文消息的 `IllegalArgumentException`。本次继续沿用现有响应处理方式，不扩大全局异常改造范围。

### 前端运行环境

`package.json` 增加 Node.js 版本约束，要求 Node 20 或更高版本。运行文档同步更新，避免 Node 14 对 `??=` 语法不支持造成构建过程假成功。

## 数据流

### 注册和登录

```text
注册请求 → 参数校验 → BCrypt 哈希 → 写入用户表
登录请求 → 查询用户 → BCrypt matches → 生成 JWT → 返回前端
```

### 健康数据保存

```text
前端保存 → POST /api/health/record → JWT 获取 userId
→ 校验日期和数据 → 查询当日记录 → 新增或更新
→ 数据库唯一索引保证同日唯一
```

### 健康统计

```text
前端请求 days → 校验 1..365 → 查询日期范围
→ 每项指标过滤缺失值 → 计算平均值 → 生成趋势数组
```

## 错误处理

- BCrypt 登录失败继续返回现有“密码错误”业务消息。
- 参数越界返回具体错误信息，例如“统计天数必须在 1 到 365 之间”。
- 导出格式无效时不再默认回退为 JSON，而是明确拒绝。
- AI 服务现有降级逻辑保持不变。
- 数据库唯一约束冲突不作为正常业务路径；正常请求应由已有记录更新逻辑处理。

## 测试策略

采用测试驱动方式补充后端测试：

1. `UserService`：注册产生 BCrypt 哈希、正确密码可登录、错误密码被拒绝。
2. `HealthDataService`：缺失指标不参与平均值、完全缺失返回 0、无效 days 被拒绝。
3. `HealthDataService`：分页参数边界和导出格式校验。

Mapper 使用 Mockito 隔离数据库，使单元测试可以在没有 MySQL 的环境中运行。数据库唯一约束通过 schema 检查和后端完整构建验证。

前端本次不引入新的测试框架；通过 TypeScript/Taro 构建验证接口调用和配置变更。

## 验收标准

- 后端 `mvn test` 通过，并实际执行新增测试。
- 新注册用户密码字段为 BCrypt 哈希，明文和 MD5 均不会写入数据库。
- 健康统计不会因为未填写指标而稀释平均值。
- 前端不再引用 `/health/record/{id}`。
- schema 中存在 `(user_id, record_date)` 唯一索引。
- `application.yml` 不再固定写入实际数据库密码、JWT 密钥和 AI Key。
- Node 20+ 环境下微信小程序构建成功。
